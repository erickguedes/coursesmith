# CourseSmith Domain Model

Version: 1.0

Status: Draft

---

# Purpose

This document defines the conceptual model used by CourseSmith.

It describes the educational entities manipulated by the framework and the relationships between them.

The Runtime MUST use these concepts consistently.

Plugins MUST NOT redefine them.

---

# Design Philosophy

CourseSmith is **Artifact-Centric**.

Every piece of information generated, consumed or transformed by the framework is represented as an Artifact.

Artifacts are immutable outputs produced by agents.

Agents never exchange natural language.

Agents exchange Artifacts.

---

# Core Abstraction

Everything in CourseSmith derives from a single concept.

```text
Artifact
```

Every entity generated by the framework is a specialization of Artifact.

Examples:

```text
Artifact

├── Course
├── Module
├── Lesson
├── Lesson Plan
├── Quiz
├── Exercise
├── Project
├── Flashcards
├── Summary
├── Diagram
├── References
├── Metadata
└── Manifest
```

This guarantees that every plugin understands the same language.

---

# Artifact

## Definition

An Artifact is a structured output produced by exactly one agent.

Artifacts may be consumed by zero or more downstream agents.

Artifacts MUST be serializable.

Preferred formats:

* Markdown
* JSON
* YAML

---

## Properties

Every Artifact contains:

```yaml
id:
type:
version:
title:
createdBy:
createdAt:
schemaVersion:
metadata:
```

Optional:

```yaml
dependsOn:
tags:
language:
references:
```

# Course

Represents the highest educational unit.

Contains:

* Metadata
* Modules
* Manifest
* Global Objectives

Produces:

```text
course.yaml
```

---

# Module

A logical division of a course.

Contains:

* Lessons
* Objectives
* Summary
* Projects

Produces:

```text
module.yaml
```

---

# Lesson

The primary educational unit.

Every lesson is independent.

Contains:

* Learning Objectives
* Topics
* Explanation
* Examples
* Practical Section
* Exercises
* Summary
* References

Produces:

```text
lesson.md
```

---

# Topic

A conceptual subdivision of a Lesson.

A Topic represents one specific concept.

Topics should be small enough to explain clearly.

Large Topics should be split.

---

# Learning Objective

Represents measurable learning outcomes.

Every lesson should contain one or more objectives.

Example:

* Explain Docker networking
* Configure Kubernetes ingress
* Understand TCP/IP fundamentals

---

# Example

Illustrates a concept.

Examples should be:

* realistic
* contextual
* concise

---

# Exercise

Allows learners to practice.

Exercise types:

* theoretical
* practical
* coding
* reflection
* research

---

# Quiz

Evaluates understanding.

Question types:

* multiple choice

* true or false

* open answer

* scenario analysis

Each question contains:

Question

Answer

Explanation

Difficulty

---

# Lab

Represents a guided practical activity.

Structure:

Objective

Prerequisites

Steps

Expected Result

Validation

Cleanup

---

# Project

Represents a larger practical assignment.

Projects consolidate knowledge across multiple lessons.

Projects may belong to:

Lesson

Module

Course

---

# Flashcards

Learning reinforcement artifacts.

Each Flashcard contains:

Front

Back

Tags

Difficulty

---

# Summary

Condensed version of a lesson.

Maximum size should remain configurable.

---

# Diagram

Visual explanation.

Preferred syntax:

Mermaid

PlantUML (future)

---

# Reference

External learning material.

Examples:

Books

Articles

RFCs

Official documentation

Videos

Research papers

---

# Manifest

Represents the generated project.

Contains:

Course metadata

Artifact list

Dependencies

Pipeline version

Plugin versions

Output checksum

Produces:

```text
manifest.json
```

# Dependency Rules

Lessons belong to one Module.

Modules belong to one Course.

Exercises belong to one Lesson.

Quizzes belong to one Lesson.

Labs belong to one Lesson.

Projects may aggregate multiple Lessons.

---

# Artifact Lifecycle

Every Artifact passes through the following lifecycle.

```text
Created

↓

Validated

↓

Published

↓

Consumed

↓

Archived
```

No Artifact should be modified in place.

New versions generate new Artifacts.

---

# Identity

Every Artifact receives a unique identifier.

Example:

```text
lesson-networking-001

quiz-networking-001

lab-networking-001
```

Identifiers remain stable.

---

# Versioning

Every Artifact carries its own version.

Example:

```yaml
version: 1.0.0
```

Version changes must never overwrite previous versions.

---

# Validation

Artifacts should validate against JSON Schemas.

Validation occurs before publication.

Invalid Artifacts stop the pipeline.

---

# Domain Rules

A Lesson MUST belong to exactly one Module.

A Module MUST belong to exactly one Course.

A Lesson MUST contain at least one Learning Objective.

A Quiz MUST reference a Lesson.

A Summary MUST reference a Lesson.

A Lab MUST reference a Lesson.

A Project MUST reference one or more Lessons.

---

# Future Artifact Types

Future versions may introduce new Artifacts without modifying the Core.

Examples:

Podcast Script

Slide Deck

Interactive Simulation

Audio Lesson

Video Script

Assessment

Mind Map

Knowledge Graph

Presentation

Certification Exam

The Runtime should remain unaware of these specific types.

It only recognizes Artifacts.

---

# Summary

The Domain Model establishes a universal language shared by every component of CourseSmith.

All plugins, agents, pipelines and publishers MUST operate exclusively on Artifacts.

This abstraction guarantees extensibility, interoperability and long-term maintainability.
